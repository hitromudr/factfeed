# Архитектура проекта "Сортировочная" (The Sorter / FactFeed)

Документ описывает внутреннее устройство, ключевые компоненты и поток данных (data flow) приложения.

## Высокоуровневая архитектура

Проект построен по модульному принципу (Монолит с четким разделением слоев) с использованием современного асинхронного стека Python.

1.  **Frontend (UI)**: Server-Side Rendering (Jinja2) + HTMX.
2.  **Backend (API & Web)**: FastAPI.
3.  **Database**: PostgreSQL 16 (с полнотекстовым поиском GIN).
4.  **Ingestion Pipeline**: Автономный планировщик задач (APScheduler) для сбора данных.
5.  **NLP Engine**: Обработка естественного языка (`spaCy` + `transformers`).
6.  **Translation Service**: Асинхронный кэширующий слой перевода.

## Ключевые компоненты

### 1. Ingestion Pipeline (Сбор данных)
Расположение: `factfeed/ingestion/`

Отвечает за регулярный сбор новостей. Запускается в фоновом потоке FastAPI.
*   **scheduler.py**: Управляет расписанием (каждые 15 минут).
*   **runner.py**: Оркестратор. Собирает RSS-фиды, проверяет дубликаты, запускает скачивание.
*   **fetcher.py**: Загрузчик. Использует `curl_cffi` для обхода TLS-фингерпринтинга и anti-bot систем. Поддерживает ротацию прокси-туннелей (Riga, Polka, Turka, Hitro).
*   **extractor.py**: Извлекает чистый текст и HTML-тело из веб-страниц с помощью `trafilatura` и `readability-lxml`.

### 2. NLP Pipeline (Анализ текста)
Расположение: `factfeed/nlp/`

Разбивает текст на предложения и классифицирует их по шкале "Факт / Мнение".
*   **segmenter.py**: Использует `spaCy` для корректного разбиения длинного текста на предложения, сохраняя пунктуацию и переносы строк.
*   **pre_filter.py**: Быстрый эвристический фильтр на основе регулярных выражений. Отсеивает "мусор" (<8 слов) и явно размечает цитаты/атрибуции ("Трамп заявил...") как `mixed` (Смешанный тип).
*   **classifier.py**: Нейросетевой классификатор (Zero-Shot). Использует легковесную многоязычную модель `mDeBERTa-v3`. Запускается только для предложений, не отсеянных пре-фильтром.
*   **calibrator.py**: (`TemperatureScaler`) Смягчает излишнюю уверенность нейросети (Temperature Scaling), превращая жесткие 99% в более реалистичные оценки.

### 3. Translation Layer (Модуль перевода)
Расположение: `factfeed/nlp/translator.py`

Поскольку базовая NLP-классификация и исходники преимущественно англоязычные (или обрабатываются мультиязычной моделью), пользовательский UI локализуется "на лету".
*   **Логика**: При запросе статьи с `lang=ru`, система переводит контент (заголовок, тело) и отдельные предложения.
*   **Кэширование**: Результаты перевода сохраняются в таблицу `translations` (в колонке `sentences_data` в формате JSON). При повторном запросе API перевода не дергается.
*   **Устойчивость**: Добавлены таймауты (5 секунд). При падении сервиса перевода текст элегантно деградирует до языка оригинала без "подвисания" UI.

### 4. Web & API (Веб-слой)
Расположение: `factfeed/web/`

*   **routes/**: UI-контроллеры, рендерящие HTML через Jinja2. Поддерживают HTMX (возвращают фрагменты HTML для AJAX).
*   **api/v1/**: REST API для интеграций (JSON).
*   **services/system_monitor.py**: In-memory синглтон, отслеживающий состояние парсера (Live Pipeline). Отвечает за зеленый виджет в шапке.

## Поток данных (Data Flow)

### Сценарий 1: Фоновый сбор новостей
1. `APScheduler` вызывает `run_ingestion_cycle()`.
2. `SystemMonitor` меняет статус на `Live Pipeline`.
3. `fetcher` асинхронно скачивает XML всех источников.
4. Для каждого нового URL: `fetcher` загружает HTML (через proxy-ротацию) -> `extractor` достает текст -> сохраняется в БД (`Article`).
5. После цикла сбора: все новые статьи, у которых нет предложений, отправляются в `NLP Pipeline`.
6. Текст -> `segmenter` -> `pre_filter` -> `classifier` -> сохранение `sentences` в БД.

### Сценарий 2: Чтение статьи пользователем (с переводом)
1. Пользователь запрашивает `/article/123`. Возвращается базовая структура (шапка, футер) и запускается HTMX-запрос на `/article/123/content?lang=ru`.
2. Если статья короткая (менее 6 предложений), она переводится целиком.
3. Система запрашивает переводы из кэша (БД: `translations`).
4. Для непереведенных предложений запускается `translate_text` (асинхронно).
5. После перевода новые строки сохраняются в БД.
6. HTML (`_article_body.html`) генерируется с применением CSS-классов (зеленый/желтый/синий фон) на основе `label` из БД.

### Сценарий 3: "Принудительно обновить" (Force Sync)
1. Пользователь кликает на кнопку обновления в UI.
2. Бэкенд вызывает `ingest_article_on_demand(force=True)`.
3. Статья принудительно перекачивается (игнорируя статус `is_partial`).
4. Мгновенно (в том же HTTP-запросе) статья прогоняется через NLP-классификатор.
5. Старый кэш переводов для этой статьи **сбрасывается**.
6. Пользователь перенаправляется обратно, получая свежий, размеченный текст.

## База Данных (ER Schema)

*   `sources`: Источники новостей (название, RSS URL, код страны).
*   `articles`: Основная таблица (url, hash, title, body, published_at, is_partial, tsv vector для поиска).
*   `sentences`: Разметка предложений (article_id, position, text, label: 'fact'|'opinion'|'mixed', confidence).
*   `translations`: Кэш локализации (article_id, language, title, body, sentences_data JSON).
