## Decisions Made

- Hand-written migration zero is the ground truth — autogenerate is disabled for initial migration because PostgreSQL `GENERATED ALWAYS AS STORED` syntax is not expressible via `op.add_column`
- GIN index exclusion uses `include_name` lambda in `context.configure` — prevents Alembic from detecting false-positive "index changed" on every subsequent `alembic revision --autogenerate`
- `search_vector` column added via `op.execute(ALTER TABLE ... ADD COLUMN ...)` — the only way to use PostgreSQL-specific generated column syntax through Alembic
- Docker Compose uses native health conditions (`service_healthy`, `service_completed_successfully`) — no shell scripts, no `wait-for-it.sh`
- `restart: on-failure` on migrate service — belt-and-suspenders retry if postgres readiness race condition occurs despite healthcheck
- `--workers 1` on app uvicorn command — APScheduler must run in single worker (set at infrastructure level to prevent future misconfiguration)

## The GIN Index Autogenerate Workaround

PostgreSQL returns the GIN index expression with internal type casts (e.g., `to_tsvector('english'::regconfig, ...)`) that differ from what SQLAlchemy/Alembic emits when comparing. This causes Alembic to detect the index as "changed" on every `autogenerate` run, even when nothing has changed.

The fix is in `alembic/env.py` `do_run_migrations()`:

```python
include_name=lambda name, type_, parent_names: (
    name not in {"ix_articles_search_vector"} if type_ == "index" else True
),
```

This tells Alembic to skip the named index entirely during schema comparison. The index is managed exclusively through hand-written migrations.

## Docker Compose Startup Chain

```
postgres (pg_isready healthcheck passes)
    |
    v
migrate (alembic upgrade head — exits 0)
    |
    v
app (uvicorn starts — database guaranteed to have schema)
```

- `postgres` has a `healthcheck` using `pg_isready` (10s start_period, 5s interval, 10 retries)
- `migrate` has `depends_on: postgres: condition: service_healthy` and `restart: on-failure`
- `app` has `depends_on: postgres: condition: service_healthy` AND `migrate: condition: service_completed_successfully`
