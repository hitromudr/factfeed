## Summary

Phase 1 establishes the PostgreSQL schema and all supporting infrastructure (project scaffolding, Docker Compose, Alembic migrations) before any application logic is written. The schema has five non-negotiable elements baked in from the start: the `tsvector GENERATED ALWAYS AS ... STORED` column with GIN index on `articles`, the `sentences` child table (not a JSON column), `url_hash` with a UNIQUE constraint, and proper startup ordering in Docker Compose.

The stack is fully verified: SQLAlchemy 2.0's `Computed` construct handles the generated column, Alembic 1.18.4 initialized with `-t async` handles async migrations against asyncpg, and Docker Compose's `condition: service_completed_successfully` on the `migrate` service handles ordering. The critical implementation risk is Alembic's known false-positive detection of changes on tsvector indexes — the workaround (excluding the GIN index from autogenerate comparison by name) must be applied in `env.py` from the start.

The project scaffold uses a flat layout (not `src/`) since `factfeed/` is a single package placed at the repository root, managed by uv with `uv init --package`. pydantic-settings reads `DATABASE_URL` and other config from `.env` via `SettingsConfigDict(env_file=".env")`.

**Primary recommendation:** Write the single initial Alembic migration by hand (not autogenerated) to ensure the tsvector GENERATED column and GIN index are created exactly right, then disable autogenerate comparison on that index by name to prevent false drift detection.

### Pattern 1: Async SQLAlchemy Engine + Session Factory

**What:** Create a single async engine and session factory in `factfeed/db/session.py`. All database access goes through this factory via dependency injection.

**When to use:** All database access in the project.

**Example:**
```python
# factfeed/db/session.py
# Source: https://docs.sqlalchemy.org/en/20/orm/extensions/asyncio.html
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker, AsyncSession
from factfeed.config import settings

engine = create_async_engine(
    settings.database_url,
    echo=settings.debug,
    pool_pre_ping=True,
)

AsyncSessionLocal = async_sessionmaker(
    engine,
    class_=AsyncSession,
    expire_on_commit=False,
)
```

### Pattern 2: Alembic Async env.py

**What:** Alembic cannot use async natively — it bridges via `connection.run_sync()`. The `-t async` template generates this pattern.

**When to use:** All async SQLAlchemy projects with Alembic.

**Example:**
```python
# alembic/env.py (key async section)
# Source: https://alembic.sqlalchemy.org/en/latest/cookbook.html#using-asyncio-with-alembic
import asyncio
from sqlalchemy.ext.asyncio import async_engine_from_config
from sqlalchemy import pool
from alembic import context

def do_run_migrations(connection):
    context.configure(
        connection=connection,
        target_metadata=target_metadata,
        # CRITICAL: exclude the GIN index name from autogenerate comparison
        # to prevent false "index changed" drift detection (Alembic issue #1390)
        include_name=lambda name, type_, parent_names: (
            name not in {"ix_articles_search_vector"} if type_ == "index" else True
        ),
    )
    with context.begin_transaction():
        context.run_migrations()

async def run_async_migrations():
    connectable = async_engine_from_config(
        config.get_section(config.config_ini_section),
        prefix="sqlalchemy.",
        poolclass=pool.NullPool,
    )
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await connectable.dispose()

def run_migrations_online():
    asyncio.run(run_async_migrations())
```

### Pattern 3: SQLAlchemy Model with tsvector GENERATED Column

**What:** Use SQLAlchemy's `Computed` construct with `persisted=True` to declare the `GENERATED ALWAYS AS ... STORED` column. PostgreSQL computes this automatically on INSERT/UPDATE.

**When to use:** The `articles` table `search_vector` column.

**Example:**
```python
# factfeed/db/models.py
# Source: https://docs.sqlalchemy.org/en/20/core/defaults.html#computed-ddl
from sqlalchemy import Column, Integer, String, Text, DateTime, Float, ForeignKey, Computed, Index
from sqlalchemy.dialects.postgresql import TSVECTOR
from sqlalchemy.orm import DeclarativeBase, relationship
import hashlib

class Base(DeclarativeBase):
    pass

class Source(Base):
    __tablename__ = "sources"
    id = Column(Integer, primary_key=True)
    name = Column(String(100), nullable=False)
    feed_url = Column(Text, nullable=False, unique=True)
    articles = relationship("Article", back_populates="source")

class Article(Base):
    __tablename__ = "articles"
    id = Column(Integer, primary_key=True)
    url = Column(Text, nullable=False)
    url_hash = Column(String(64), nullable=False, unique=True)  # sha256 hex
    title = Column(Text, nullable=False)
    body = Column(Text, nullable=True)
    published_at = Column(DateTime(timezone=True), nullable=True)
    source_id = Column(Integer, ForeignKey("sources.id"), nullable=True)
    # GENERATED ALWAYS AS STORED — PostgreSQL maintains this automatically
    search_vector = Column(
        TSVECTOR,
        Computed(
            "to_tsvector('english', coalesce(title, '') || ' ' || coalesce(body, ''))",
            persisted=True,
        ),
        nullable=True,
    )
    source = relationship("Source", back_populates="articles")
    sentences = relationship("Sentence", back_populates="article",
                             cascade="all, delete-orphan", order_by="Sentence.position")

    __table_args__ = (
        # GIN index defined here; excluded from Alembic autogenerate by name (see env.py)
        Index("ix_articles_search_vector", "search_vector", postgresql_using="gin"),
    )

class Sentence(Base):
    __tablename__ = "sentences"
    id = Column(Integer, primary_key=True)
    article_id = Column(Integer, ForeignKey("articles.id", ondelete="CASCADE"), nullable=False)
    position = Column(Integer, nullable=False)       # sentence order within article
    text = Column(Text, nullable=False)
    label = Column(String(20), nullable=True)        # fact / opinion / mixed / unclear
    confidence = Column(Float, nullable=True)        # 0.0 – 1.0
    article = relationship("Article", back_populates="sentences")
```

### Pattern 4: Hand-Written Initial Alembic Migration

**What:** Write `0001_initial_schema.py` entirely by hand. Autogenerate does not reliably produce `GENERATED ALWAYS AS ... STORED` syntax or the correct `CREATE INDEX ... USING GIN` clause. The hand-written migration is the ground truth.

**When to use:** For the initial migration containing generated columns and custom index types.

**Example:**
```python
# alembic/versions/0001_initial_schema.py
# Source: https://www.postgresql.org/docs/current/textsearch-tables.html
"""Initial schema: articles, sentences, sources

Revision ID: 0001
"""
from alembic import op
import sqlalchemy as sa

def upgrade():
    op.create_table(
        "sources",
        sa.Column("id", sa.Integer(), primary_key=True),
        sa.Column("name", sa.String(100), nullable=False),
        sa.Column("feed_url", sa.Text(), nullable=False, unique=True),
    )

    op.create_table(
        "articles",
        sa.Column("id", sa.Integer(), primary_key=True),
        sa.Column("url", sa.Text(), nullable=False),
        sa.Column("url_hash", sa.String(64), nullable=False),
        sa.Column("title", sa.Text(), nullable=False),
        sa.Column("body", sa.Text(), nullable=True),
        sa.Column("published_at", sa.DateTime(timezone=True), nullable=True),
        sa.Column("source_id", sa.Integer(), sa.ForeignKey("sources.id"), nullable=True),
    )
    op.create_unique_constraint("uq_articles_url_hash", "articles", ["url_hash"])

    # GENERATED ALWAYS AS STORED — must be raw SQL; no SQLAlchemy op equivalent
    op.execute("""
        ALTER TABLE articles
        ADD COLUMN search_vector tsvector
        GENERATED ALWAYS AS (
            to_tsvector('english', coalesce(title, '') || ' ' || coalesce(body, ''))
        ) STORED
    """)

    # GIN index on the generated column
    op.execute("""
        CREATE INDEX ix_articles_search_vector ON articles USING GIN (search_vector)
    """)

    op.create_table(
        "sentences",
        sa.Column("id", sa.Integer(), primary_key=True),
        sa.Column("article_id", sa.Integer(),
                  sa.ForeignKey("articles.id", ondelete="CASCADE"), nullable=False),
        sa.Column("position", sa.Integer(), nullable=False),
        sa.Column("text", sa.Text(), nullable=False),
        sa.Column("label", sa.String(20), nullable=True),
        sa.Column("confidence", sa.Float(), nullable=True),
    )
    op.create_index("ix_sentences_article_id", "sentences", ["article_id"])

def downgrade():
    op.drop_table("sentences")
    op.drop_table("articles")
    op.drop_table("sources")
```

### Anti-Patterns to Avoid

- **Running Alembic inside the FastAPI startup event:** Fails under multiple replicas and is harder to debug. Use the dedicated `migrate` container.
- **Autogenerating the initial migration:** Alembic autogenerate cannot produce `GENERATED ALWAYS AS ... STORED` correctly. Always hand-write the first migration.
- **Defining `search_vector` as a regular column and populating it with triggers:** PostgreSQL `GENERATED ALWAYS AS ... STORED` is cleaner and does not require a separate trigger function.
- **Using `psycopg2` as the driver:** Does not support async; SQLAlchemy async engine requires `asyncpg` or `psycopg3[binary]`.
- **Using `src/` layout:** The CONTEXT.md specifies `factfeed/` as a top-level package (flat layout). Do not nest it under `src/`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Migration versioning and ordering | Custom migration runner | Alembic | Handles dependency chains, up/down migrations, multi-branch merges |
| tsvector column maintenance | Trigger function in SQL | `GENERATED ALWAYS AS ... STORED` | PostgreSQL handles updates automatically; no trigger code to maintain |
| Docker service startup ordering | Shell `wait-for-it.sh` scripts | Docker Compose `condition: service_healthy` + `condition: service_completed_successfully` | Native Compose feature; no extra scripts |
| Configuration from env | `os.environ.get()` calls | pydantic-settings `BaseSettings` | Type coercion, validation, `.env` support, IDE completion — all free |
| Async session management | Manual `await session.close()` | SQLAlchemy `async_sessionmaker` + context manager | Exception-safe teardown; handles rollback automatically |

**Key insight:** All complexity in this phase is in infrastructure plumbing, not business logic. Using the right tools (Alembic, Compose health checks, pydantic-settings) eliminates entire categories of bugs before any application code is written.

## Common Pitfalls

### Pitfall 1: Alembic False-Positive tsvector Index Drift

**What goes wrong:** After the initial migration runs, every subsequent `alembic revision --autogenerate` detects the GIN index on `search_vector` as "changed" and generates a migration to drop and recreate it — even though nothing changed.

**Why it happens:** Alembic compares index expressions as strings. PostgreSQL returns `to_tsvector('english'::regconfig, ...)` with type casts; Alembic's model definition omits them. The string comparison fails, triggering a false diff. (Open issue as of June 2024: https://github.com/sqlalchemy/alembic/issues/1390)

**How to avoid:** In `alembic/env.py`, exclude the GIN index by name from autogenerate comparison using the `include_name` hook in `context.configure()`. The initial migration creates the index via raw SQL (`op.execute`); subsequent autogenerate will ignore it.

**Warning signs:** `alembic revision --autogenerate` produces a migration with only `drop_index` + `create_index` on `ix_articles_search_vector`.

---

### Pitfall 2: `GENERATED ALWAYS AS` Column Cannot Accept Explicit Values on INSERT

**What goes wrong:** Any INSERT or UPDATE that attempts to set a value for a `GENERATED ALWAYS AS ... STORED` column raises `ERROR: column "search_vector" can only be updated to DEFAULT`.

**Why it happens:** PostgreSQL strictly enforces the GENERATED constraint. The column is server-side computed only.

**How to avoid:** Never include `search_vector` in INSERT/UPDATE statements. SQLAlchemy `Computed` columns are automatically treated as server defaults — SQLAlchemy will not include them in generated INSERT/UPDATE SQL.

**Warning signs:** `psycopg.errors.GeneratedAlways` exception on article insert. Usually caused by calling `session.execute(insert(Article).values(search_vector=...))` explicitly.

---

### Pitfall 3: asyncpg vs Alembic Sync Migration Mismatch

**What goes wrong:** Running `alembic upgrade head` against a `postgresql+asyncpg://` connection string fails because Alembic's migration runner expects a synchronous connection by default.

**Why it happens:** asyncpg only provides an async API. Alembic requires explicit async bridging via `-t async` template and `connection.run_sync()`.

**How to avoid:** Always initialize Alembic with `alembic init -t async alembic`. This generates an `env.py` that uses `async_engine_from_config` and `run_sync()` properly. Verify the `sqlalchemy.url` in `alembic.ini` uses `postgresql+asyncpg://`.

**Warning signs:** `ModuleNotFoundError: No module named 'asyncpg'` or `sqlalchemy.exc.InvalidRequestError` on `alembic upgrade head`.

---

### Pitfall 4: Docker Compose `depends_on` Without Health Conditions

**What goes wrong:** Using plain `depends_on: [postgres]` (no condition) starts the migrate container the instant the PostgreSQL container starts — before PostgreSQL is actually accepting connections. Alembic fails with `connection refused`.

**Why it happens:** Without `condition: service_healthy`, Docker Compose only waits for the container to start, not for the process inside to be ready.

**How to avoid:** Always pair PostgreSQL with a `healthcheck` using `pg_isready`. Set `depends_on.postgres.condition: service_healthy` on the migrate service. Set `depends_on.migrate.condition: service_completed_successfully` on the app service.

**Warning signs:** `migrate` container exits immediately with connection error on first `docker compose up`.

---

### Pitfall 5: Missing `asyncio_mode = "auto"` in pytest Config

**What goes wrong:** `async def` test functions fail with `PytestUnraisableExceptionWarning` or simply never run, reporting "coroutine was never awaited."

**Why it happens:** pytest-asyncio 0.25+ requires explicit mode configuration. The default mode changed from `auto` to `strict` in recent releases.

**How to avoid:** Add `asyncio_mode = "auto"` to `[tool.pytest.ini_options]` in `pyproject.toml`. This enables automatic event loop handling for all `async def` test functions.

**Warning signs:** Tests appear to pass (zero assertions run) or raise coroutine-not-awaited warnings.

---

### Pitfall 6: GIN Index Missing From Migration Zero

**What goes wrong:** The GIN index is created in a later migration or added manually. The first run of `alembic upgrade head` on a fresh database creates the `articles` table without the index. If any data is inserted before the index is added, creating the index later requires a table lock and full index rebuild — expensive on large tables.

**Why it happens:** Index creation is often treated as an optimization step, not a schema requirement.

**How to avoid:** The GIN index MUST be created in the same migration (`0001_initial_schema.py`) that creates the `articles` table, immediately after adding the `search_vector` column. Per the success criteria, the index must be "present from migration zero."

**Warning signs:** `EXPLAIN ANALYZE` on a search query shows `Seq Scan` instead of `Bitmap Index Scan using ix_articles_search_vector`.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Trigger function to maintain tsvector | `GENERATED ALWAYS AS ... STORED` column | PostgreSQL 12 (2019) | No trigger SQL; cleaner schema; maintained by DB engine |
| `@app.on_event("startup")` | `@asynccontextmanager` lifespan | FastAPI 0.93+ (2023) | `on_event` deprecated; lifespan is the correct hook for startup/shutdown |
| `session.query(Model)` ORM style | `select(Model)` SQLAlchemy 2.0 style | SQLAlchemy 2.0 (2023) | Legacy 1.x style not compatible with async engine; 2.0 style mandatory |
| `from pydantic import BaseSettings` | `from pydantic_settings import BaseSettings` | Pydantic v2 (2023) | `BaseSettings` moved to separate `pydantic-settings` package |
| `wait-for-it.sh` entrypoint scripts | Docker Compose `condition: service_healthy` | Compose v2 (2022) | Native Compose feature eliminates shell script dependency |
| pip + requirements.txt | uv + pyproject.toml + lockfile | 2024 | 10-100x faster installs; lockfile ensures reproducibility |

**Deprecated/outdated:**
- `alembic init` (without `-t async`): Generates a synchronous `env.py` that cannot work with asyncpg. Use `alembic init -t async alembic` instead.
- `APScheduler 4.x alpha`: Breaking API changes; use 3.11.2 stable.

## Open Questions

1. **Alembic autogenerate for future migrations**
   - What we know: The initial migration is hand-written. The GIN index on `search_vector` is excluded from autogenerate via `include_name` hook.
   - What's unclear: Whether future Alembic releases will fix the tsvector false-diff issue (https://github.com/sqlalchemy/alembic/issues/1390 open as of June 2024).
   - Recommendation: Treat the `include_name` filter as permanent. Document it in the migration README. If Alembic fixes the issue in a future release, it's easy to remove.

2. **Test database for pytest: real PostgreSQL vs aiosqlite**
   - What we know: `tsvector GENERATED ALWAYS AS ... STORED` and `CREATE INDEX ... USING GIN` are PostgreSQL-specific DDL. They do not exist in SQLite.
   - What's unclear: Whether CI will have a PostgreSQL 16 container available.
   - Recommendation: Use real PostgreSQL (Docker) for integration tests that validate the schema. Provide a `docker-compose.test.yml` or document `TEST_DATABASE_URL` env variable. Do NOT use aiosqlite for schema tests — the generated column and GIN index will be silently skipped or error.

3. **url_hash normalization strategy**
   - What we know: Architecture research specifies `sha256(url.strip().lower().split("?")[0])` — strip query params and lowercase.
   - What's unclear: Whether all 5 target RSS sources (BBC, Reuters, AP, NPR, Al Jazeera) use stable, deterministic URLs or whether URLs include session tokens or tracking parameters.
   - Recommendation: Implement the simple strip+lowercase normalization first. Document the exact normalization function as a constant. This can be refined in Phase 2 when actually testing against the feeds.
